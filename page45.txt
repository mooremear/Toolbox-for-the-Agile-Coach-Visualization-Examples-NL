Taakgrootte < 1 dag

Tijdens het “taken” (dwz het afpellen van het werk wat gedurende de sprint dient te gebeuren) stop je niet eerder met splitsen tot elke taak afgerond kan worden binnen een dag.

Deze afspraak zorgt ervoor dat je voortgang en bottlenecks sneller ontdekt worden. Het leidt er ook toe dat er meer wordt samengewerkt tijdens de planning sessies.

Als je nu elke dag dat een taak op “in progress” hangt een stipje op de task zet valt het meteen op als er een taak “vast” zit. Immers een taak kan niet meer dan een stipje verzamelen.

Ook kun je de sprint burndown doen in taken, dit gaat sneller dan uren en geeft een goede indicatie of alle taken af komen binnen de sprint.

Task size < 1 day

When tasking (i.e. breaking down the work that needs to be performed to complete a User Story) don’t stop splitting the tasks until each task can be completed within a day.

This policy makes it easier to see progress and bottlenecks. It also triggers deeper collaborative discussions during the planning sessions.

Furthermore, if you use Dotting, you know there is something fishy going on if a task starts collecting dots since no single task should have more than one dot.

If you use a Sprint Burndown, you can instead of summarizing hours summarize the number of tasks. Much faster!
